## 评审清单

**代码质量：**
- 职责分离是否清晰？
- 错误处理是否妥当？
- 类型安全（如适用）？
- 是否遵循了 DRY 原则？
- 边缘情况是否已处理？

**架构设计：**
- 设计决策是否合理？
- 是否考虑了可扩展性？
- 对性能有何影响？
- 是否存在安全隐患？

**测试验证：**
- 测试是否真正测试了逻辑（而非仅仅 Mock 行为）？
- 边缘情况是否覆盖？
- 是否在需要的地方添加了集成测试？
- 所有测试是否通过？

**需求吻合度：**
- 是否满足了计划中的所有需求？
- 实现方案是否匹配规范？
- 是否存在范围偏移 (Scope creep)？
- 破坏性变更 (Breaking changes) 是否已记录？

**生产环境就绪度：**
- 迁移策略（如果涉及 Schema 变更）？
- 是否考虑了向后兼容性？
- 文档是否完整？
- 是否存在显而易见的 Bug？

## 关键守则

**必须做到：**
- 按实际严重程度分类（并非所有问题都是“致命”的）。
- 保持具体（指明“文件:行号”，不要含糊其辞）。
- 解释**为什么**这些问题很重要。
- 认可优点。
- 给出明确的裁定。

**严禁行为：**
- 在未仔细检查前说“看起来不错”。
- 将细微的挑刺标记为“致命”。
- 对你没有评审的代码发表意见。
- 给出含糊的反馈（如“改进错误处理”）。
- 回避给出明确的裁定。
