# 完成前验证 (Verification Before Completion)

## 概述

在没有验证的情况下声称工作已完成是欺骗，而不是效率。

**核心原则：** 始终坚持“证据先于断言”。

**违反该规则的字面意思即是违反该规则的精神。**

## 铁律

```
没有最新的验证证据，绝不声称已完成
```

如果你没有在当前消息中运行验证命令，你就不能声称它已通过。

## 门槛自检 (Gate Function)

```
在声称任何状态或表达满意度之前：

1. 识别：哪个命令能证明这个断言？
2. 运行：执行完整的命令（最新的、全量的）。
3. 阅读：完整阅读输出，检查退出码，统计失败数量。
4. 验证：输出是否证实了断言？
   - 如果“否”：陈述真实状态并附带证据。
   - 如果“是”：陈述断言的同时附带证据。
5. 只有在完成上述步骤后：才能提出断言。

跳过任何步骤 = 撒谎，而非验证。
```

## 常见失败场景

| 断言 | 需要满足 | 不足以为据 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 失败 | 上次的运行结果，“应该会通” |
| Linter 清洁 | Linter 输出：0 错误 | 部分检查，推测 |
| 构建成功 | 构建命令：退出码为 0 | Linter 通过，日志看起来还行 |
| Bug 已修复 | 测试原始症状：通过 | 代码已改，假设已修好 |
| 回归测试有效 | 红-绿循环已验证 | 测试仅通过了一次 |
| Agent 已完成 | VCS diff 显示了变更 | Agent 汇报说“成功” |
| 需求已达成 | 逐项核对清单 | 仅仅测试通过了 |

## 红灯信号 - 停止

- 使用“应该”、“大概”、“似乎”等词汇。
- 在验证前表达满意度（“太棒了！”、“完美！”、“完成了！”等）。
- 打算在没有验证的情况下 Commit/Push/提 PR。
- 盲目相信 Agent 的成功报告。
- 依赖部分验证。
- 产生“就这一次”的想法。
- 感到疲劳并想赶快收工。
- **任何在运行验证前暗示成功的措辞。**

## 辩解反制

| 借口 | 现实情况 |
|--------|---------|
| “现在应该行了” | 请立即**运行**验证命令 |
| “我有信心” | 信心 ≠ 证据 |
| “就这一次” | 绝无例外 |
| “Linter 通过了” | Linter 通过 ≠ 编译通过 |
| “Agent 说成功了” | 请进行独立验证 |
| “我累了” | 疲劳 ≠ 借口 |
| “部分检查就够了” | 部分检查证明不了任何东西 |
| “换个说法，规则就不适用了” | 精神高于字面意思 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] “所有测试已通过”
❌ “现在应该通了” / “看起来是对的”
```

**回归测试 (TDD 红-绿)：**
```
✅ 编写测试 -> 运行（通过） -> 撤销修复 -> 运行（必须失败） -> 恢复修复 -> 运行（通过）
❌ “我已经写了一个回归测试”（但没有经过红-绿验证）
```

**构建：**
```
✅ [运行构建] [看到：退出码 0] “构建通过”
❌ “Linter 通过了”（Linter 不检查编译错误）
```

**需求核对：**
```
✅ 重读计划 -> 创建清单 -> 验证每一项 -> 汇报缺口或完成状态
❌ “测试通过了，该阶段已完成”
```

**Agent 委派：**
```
✅ Agent 汇报成功 -> 检查 VCS diff -> 验证变更 -> 汇报真实状态
❌ 盲目相信 Agent 的报告
```

## 为什么这很重要

源自多次失败的教训：
- 伙伴说“我不相信你”——信任破裂。
- 将未定义的函数发布了出去——会导致崩溃。
- 将未达标的需求发布了出去——功能缺失。
- 在假象的完成后浪费时间 -> 导致重定向 -> 重做。
- 违反准则：“诚实是核心价值观。如果你撒谎，你将被取代。”

## 何时应用

**在以下情况之前，始终执行：**
- 任何形式的成功/完成声明。
- 任何满意度的表达。
- 对工作状态的任何正面评价。
- Commit、创建 PR、任务结算。
- 转向下一个任务。
- 委派给 Agent。

**该规则适用于：**
- 精确的短语。
- 改写或同义词。
- 暗示成功的倾向。
- 任何暗示完成/正确性的沟通。

## 总结

**验证没有捷径。**

运行命令。阅读输出。**然后**再陈述结果。

这是不可逾越的底线。
