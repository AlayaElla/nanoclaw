# 系统化调试 (Systematic Debugging)

## 概述

胡乱猜测的修复是在浪费时间，并会引入新的 Bug。仓促的补丁只会掩盖底层问题。

**核心原则：** 在尝试修复之前，**始终**先找到根因。修复症状即是失败。

**违反此流程的字面要求即是违反调试的精神。**

## 铁律

```
没有调查出根因前，绝不进行修复
```

如果你没有完成“第一阶段”，你就不具备提出修复方案的资格。

## 何时使用

适用于**任何**技术问题：
- 测试失败
- 生产环境 Bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况下使用：**
- 处于时间压力下（紧急情况会让人忍不住去瞎猜）。
- “一个快速修复”看起来非常显而易见时。
- 你已经尝试了多种修复方案但均未奏效。
- 之前的修复没有起作用。
- 你并未完全理解该问题。

**不要在以下情况下跳过：**
- 问题看起来很简单（简单的 Bug 也有其根因）。
- 你很赶时间（仓促行事必然导致返工）。
- 经理要求“立刻”修复（系统化调试比无头苍蝇般的乱撞更快）。

## 四个阶段

你**必须**在完成每一个阶段后，才能进入下一个阶段。

### 第一阶段：根因调查 (Root Cause Investigation)

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告。
   - 它们通常包含确切的解决方案。
   - 完整阅读堆栈跟踪 (Stack Trace)。
   - 记录行号、文件路径、错误代码。

2. **确保稳定复现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 它是每次都发生吗？
   - 如果无法复现 -> 收集更多数据，不要瞎猜。

3. **检查最近的变更**
   - 发生了什么可能导致此问题的变化？
   - 查看 `git diff`、最近的提交。
   - 新的依赖项、配置更改。
   - 环境差异。

4. **在多组件系统中收集证据**

   **当系统包含多个组件（如 CI -> 构建 -> 签名，或 API -> 服务 -> 数据库）时：**

   **在提出修复建议之前，添加诊断性插桩 (Diagnostic Instrumentation)：**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置的传递
     - 检查每一层的状态
   
   运行一次以收集证据，查明它在哪里断掉了。
   然后分析证据以识别出发生故障的组件。
   最后针对该特定组件进行调查。
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== 工作流中的 Secrets 可用性 ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"
   
   # 第 2 层：构建脚本
   echo "=== 构建脚本中的环境变量 ==="
   env | grep IDENTITY || echo "环境变量中没有 IDENTITY"
   
   # 第 3 层：签名脚本
   echo "=== Keychain 状态 ==="
   security list-keychains
   security find-identity -v
   
   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这能揭示：** 哪一层出现了故障（例如：Secrets -> Workflow ✓，但 Workflow -> Build ✗）。

5. **追踪数据流**

   **当错误深藏在调用栈中时：**

   参见本目录下的 `root-cause-tracing.md` 了解完整的“向后追踪”技术。

   **快速版本：**
   - 异常值起源于何处？
   - 谁用异常值调用了这里？
   - 持续向上追踪直到找到源头。
   - 在源头修复，而不是在症状处修复。

### 第二阶段：模式分析 (Pattern Analysis)

**在修复前找到规律：**

1. **寻找正常工作的示例**
   - 在同一代码库中定位类似的正常工作的代码。
   - 与损坏版本类似但运行正常的代码是什么样的？

2. **对比参考实现**
   - 如果正在实现某种模式，请完整阅读参考实现。
   - 不要略读——读每一行。
   - 在应用之前充分理解该模式。

3. **识别差异**
   - 正常的和损坏的之间有什么不同？
   - 列出每一个差异，无论多么微小。
   - 不要先入为主地认为“那无关紧要”。

4. **理解依赖关系**
   - 这需要哪些其他组件？
   - 需要哪些设置、配置、环境？
   - 它做了哪些假设？

### 第三阶段：假设与测试 (Hypothesis and Testing)

**采用科学方法：**

1. **形成单一假设**
   - 清晰陈述：“我认为 X 是根因，因为 Y”。
   - 把它写下来。
   - 要具体，不要含糊。

2. **进行最小化测试**
   - 做出能测试该假设的、尽可能小的改动。
   - 一次只改一个变量。
   - 不要同时修复多件事。

3. **继续前的验证**
   - 奏效了吗？是 -> 进入第四阶段。
   - 没奏效？形成**新的**假设。
   - **不要**在旧修复之上叠新修复。

4. **当你不知道时**
   - 坦诚说“我不理解 X”。
   - 不要不懂装懂。
   - 寻求帮助。
   - 进行更多研究。

### 第四阶段：实施 (Implementation)

**修复根因，而非症状：**

1. **创建失败的测试用例**
   - 最简单的复现方式。
   - 尽可能使用自动化测试。
   - 如果没有框架，使用一次性测试脚本。
   - 修复前**必须**拥有测试。
   - 使用 `superpowers:test-driven-development` Skill 编写标准的失败测试。

2. **实施单一修复**
   - 针对识别出的根因进行处理。
   - 一次只做一个改动。
   - 不要顺便做“既然我都改到这了”的改进。
   - 不要捆绑重构。

3. **验证修复**
   - 测试现在通过了吗？
   - 没有破坏其他测试吗？
   - 问题真的解决了吗？

4. **如果修复不起作用**
   - **停止**。
   - 计数：你已经尝试了几个修复方案？
   - 如果 < 3 个：返回第一阶段，结合新信息重新分析。
   - **如果 ≥ 3 个：停止并质疑架构（见下方第 5 步）**。
   - 在进行架构讨论前，不要尝试第 4 个修复方案。

5. **如果 3 次以上修复失败：质疑架构**

   **表明存在架构问题的迹象：**
   - 每次修复都会在不同地方揭示出新的共享状态/耦合/问题。
   - 实施修复需要进行“大规模重构”。
   - 每个修复都会在别处产生新的症状。

   **停止并质疑基本面：**
   - 这个模式从根本上说是可靠的吗？
   - 我们是不是纯粹因为“惯性”而坚持这样做？
   - 我们是应该重构架构，还是继续修复症状？

   **在尝试更多修复之前，先与人类伙伴讨论。**

   这不仅仅是假设失败——这可能是架构错误。

## 红灯信号 - 停止并遵循流程

如果你发现自己产生了以下念头：
- “先快速修复，以后再调查。”
- “试着改改 X，看看行不行。”
- “加入多处改动，运行测试。”
- “跳过测试，我会手动验证。”
- “大概是 X 的问题，让我修复它。”
- “我不完全理解，但这个可能会有用。”
- “模式说是 X，但我打算换种方式适配。”
- “这是主要问题：[在没有调查前列出修复方案]”
- 在追踪数据流之前就提出方案。
- **“再尝试最后一个修复方案”（当你已经试过 2 个以上时）。**
- **每次修复都在不同地方引出新问题。**

**所有这些都意味着：停止。回到第一阶段。**

**如果 3 次以上修复失败：** 质疑架构（参见 4.5 节）。

## 人类伙伴发出的警告信号

**注意这些重定向信号：**
- “那是没发生吗？” —— 你在没有验证的情况下做了假设。
- “它能向我们展示……吗？” —— 你应该增加证据收集环节。
- “别瞎猜” —— 你在不理解的情况下提出方案。
- “深入思考下 (Ultrathink this)” —— 质疑基本面，而不只是症状。
- “我们卡住了？”（感到沮丧） —— 你的方法不起作用。

**当你看到这些信号时：** 停止。回到第一阶段。

## 常见的辩解理由

| 借口 | 现实情况 |
|--------|---------|
| “问题很简单，不需要流程” | 简单的问题也有根因。对于简单 Bug 来说流程很快。 |
| “紧急情况，没时间走流程” | 系统化调试比“瞎试”这种乱撞**更快**。 |
| “先试试这个，然后再调查” | 第一个修复方案决定了模式。从一开始就做对。 |
| “我确认修复有效后再写测试” | 未经测试的修复站不住脚。先写测试能证明它。 |
| “一次多做几个修复能省时间” | 无法隔离出哪个起作用了。会导致新 Bug。 |
| “参考太长了，我自己适配模式” | 片面的理解必然导致 Bug。请完整阅读。 |
| “我看到问题了，让我修复它” | 看到症状 ≠ 理解根因。 |
| “再试最后一个修复方案”（失败 2 次以上后） | 3 次以上失败 = 架构问题。质疑模式，不要再修了。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因调查** | 阅读错误、复现、检查变更、收集证据 | 理解“是什么”和“为什么” |
| **2. 模式分析** | 寻找正常示例、对比 | 识别差异 |
| **3. 假设测试** | 形成理论、最小化测试 | 确认假设或形成新假设 |
| **4. 实施修复** | 创建测试、修复、验证 | Bug 解决，测试通过 |

## 当流程显示“没有根因”时

如果系统化调查显示问题确实源于环境、时序依赖或外部因素：

1. 你已完成了流程。
2. 记录你调查过的内容。
3. 实施适当的处理机制（重试、超时、错误消息）。
4. 为未来的进一步调查添加监控/日志。

**但是：** 95% 的“没有根因”案例其实都是调查不完整。

## 辅助技术

以下技术属于系统化调试的一部分，可在本目录中找到：

- **`root-cause-tracing.md`** —— 通过调用栈向后追踪 Bug，找到原始触发点。
- **`defense-in-depth.md`** —— 找到根因后，在多个层级添加校验。
- **`condition-based-waiting.md`** —— 用条件轮询取代随意的超时等待。

**相关 Skill：**
- **superpowers:test-driven-development** —— 用于创建失败的测试用例（第四阶段第 1 步）。
- **superpowers:verification-before-completion** —— 在声称成功前验证修复是否有效。

## 实际影响

源自调试会话的数据：
- 系统化方法：15-30 分钟修复。
- 胡乱修复方法：2-3 小时的乱撞。
- 首次修复成功率：95% vs 40%。
- 引入新 Bug 的几率：接近零 vs 常见。
