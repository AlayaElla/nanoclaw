# 编写计划 (Writing Plans)

## 概述

编写详尽的实施计划。假设负责执行的工程师对我们的代码库一无所知，且审美和习惯可能并不可靠。记录他们需要知道的一切：每项任务需要动哪些文件、代码实现、需要检查的测试和文档，以及如何进行测试。将整个计划拆解为“一口大小”的任务分发给他们。遵循 DRY（不写重复代码）、YAGNI（你不需要它）、TDD（测试驱动开发）以及频繁提交的原则。

假设他们是技术熟练的开发者，但对我们的工具链或业务领域几乎一无所知，且不太擅长设计优秀的测试。

**开始时请宣告：** “我正在使用 `writing-plans` Skill 来创建实施计划。”

**上下文：** 此操作应当在专用的工作树 (Worktree) 中运行（由 `brainstorming` Skill 创建）。

**建立计划：** 必须调用原生的 `EnterPlanMode` 工具进入计划模式，并利用该工具制定与确认计划。
- (请勿简单保存为普通 markdown 或静态文件，这是强制动作)

## 范围检查

如果规范涵盖了多个独立的子系统，则在头脑风暴阶段就应该将其拆分为子项目。如果没有拆分，请建议将此任务拆分为多个独立的计划 —— 每个子系统一个。每个计划都应该能够独立产生可运行、可测试的软件。

## 文件结构

在定义任务之前，勾勒出将要创建或修改哪些文件，以及每个文件负责什么。这是确定拆解决策的地方。

- 设计具有清晰边界和定义良好的接口的单元。每个文件应有且只有一个明确的职责。
- 当你能同时在上下文中推理出一段代码时，你的推理效果最好；当文件聚焦时，你的编辑也更可靠。优先选择小而聚焦的文件，而不是做得太多的庞然大物。
- 经常一起变动的文件应该放在一起。按职责划分，而不是按技术层级划分。
- 在现有代码库中，遵循既定模式。如果代码库习惯使用大文件，不要单方面重组 —— 但如果你正在修改的文件已经变得臃肿不堪，在计划中包含拆分逻辑是合理的。

这种结构贯穿了任务拆解。每个任务都应产生独立且有意义的自洽变更。

## “一口大小”的任务粒度

**每一步都是一个动作（耗时 2-5 分钟）：**
- “编写失败的测试” —— 一个步骤
- “运行并确保它失败” —— 一个步骤
- “编写最简代码使测试通过” —— 一个步骤
- “运行测试并确保它们通过” —— 一个步骤
- “提交 (Commit)” —— 一个步骤

## 计划文档头

**每个计划必须以该标题开头：**

```markdown
# [功能名称] 实施计划

> **致 Agent 执行者：** 必需调用的子 Skill：使用 `superpowers:subagent-driven-development`（推荐）或 `superpowers:executing-plans` 来逐项执行此计划。步骤使用复选框 (`- [ ]`) 语法进行进度跟踪。

**目标：** [用一句话描述构建的内容]

**架构：** [用 2-3 句话描述实现方案]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### 任务 N: [组件名称]

**涉及文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：编写失败的测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试以验证其失败**

运行：`pytest tests/path/test.py::test_name -v`
预期结果：失败，报错信息为 "function not defined"

- [ ] **步骤 3：编写最小化实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试以验证其通过**

运行：`pytest tests/path/test.py::test_name -v`
预期结果：通过 (PASS)

- [ ] **步骤 5：提交代码**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: 增加特定功能"
```
````

## 严禁占位符

每一步都必须包含工程师真正需要的实际内容。以下内容属于**计划失效** —— 绝不要写：
- "TBD"、"TODO"、“稍后实现”、“填充细节”。
- “添加适当的错误处理” / “增加校验” / “处理边缘情况”。
- “为上述内容编写测试”（而不给出实际测试代码）。
- “类似于任务 N”（请重复代码 —— 工程师可能会不按顺序阅读任务）。
- 只描述该做什么而不展示如何做的步骤（代码步骤必须包含代码块）。
- 引用未在任何任务中定义的类型、函数或方法。

## 谨记
- 始终提供确切的文件路径。
- 每一步都提供完整的代码 —— 如果某步更改了代码，请展示出来。
- 提供确切的命令及预期输出。
- 遵循 DRY、YAGNI、TDD 原则，频繁提交。

## 自测环节 (Self-Review)

在编写完完整计划后，以全新的视角审视规范并据此检查计划。这是一个你需要亲自运行的清单 —— 而不是派发给子 Agent 的。

**1. 规范覆盖率：** 略读规范中的每个部分/要求。你能指出实现了它的具体任务吗？列出任何缺口。

**2. 占位符扫描：** 在你的计划中搜索红灯信号 —— 即上述“严禁占位符”部分提到的任何模式。修复它们。

**3. 类型一致性：** 你在后续任务中使用的类型、方法签名和属性名称是否与你在之前任务中定义的相匹配？例如在任务 3 中叫 `clearLayers()` 但在任务 7 中叫 `clearFullLayers()`，这就是一个 Bug。

如果发现问题，请直接在文档中修复。无需重新审查 —— 修正后继续即可。如果发现规范要求没有对应的任务，请补上。

## 执行交付 (Execution Handoff)

保存计划后，提供执行选项：

**“计划已通过 `EnterPlanMode` 工具创建并确认。有两种执行方案：**

**1. 子 Agent 驱动 (推荐)** —— 我将为每个任务分派一个全新的子 Agent，在任务间进行评审，实现快速迭代。

**2. 行内执行** —— 在当前会话中使用 `executing-plans` 执行任务，分批执行并设有评审检查点。

**请问选择哪种方案？”**

**如果选择了“子 Agent 驱动”：**
- **必需调用的子 Skill：** 使用 `superpowers:subagent-driven-development`。
- 每个任务配备全新的子 Agent + 两阶段评审。

**如果选择了“行内执行”：**
- **必需调用的子 Skill：** 使用 `superpowers:executing-plans`。
- 分批执行并设有评审检查点。
